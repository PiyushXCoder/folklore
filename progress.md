- [x] **Fixed: desktop live-reload wasn't firing at all.** First attempt used `@tauri-apps/plugin-fs`'s `watch()` gated by a `fs:scope: ["**"]` capability — that promise silently never resolved/fired and the frontend had no `.catch`, so nothing surfaced. Replaced with a custom Rust implementation: `watch_path`/`unwatch_path` commands built directly on the `notify` crate (like `read_file_bytes`, a plain command bypasses the fs-plugin ACL instead of fighting its scope glob syntax), emitting a per-watch `file-changed:<id>` Tauri event the frontend listens for and debounces (200ms — a single save fires multiple inotify events). Removed `@tauri-apps/plugin-fs` and the `fs:*` capability entries entirely. Verified the `notify` watch logic in isolation with a standalone Rust smoke test (write → Modify event received) before wiring it back into the app.
- [x] **Fixed: live-reload only worked "sometimes."** Root cause: `watch_path` watched the file's own path directly. Editors that save atomically (write a temp file + `rename()` over the original — vim, VS Code, many others) replace the inode on save; a watch on the file's own path goes dead after the *first* such save, so plain in-place writes kept working but atomic-save editors broke after edit #1. Confirmed with a standalone test: 0 events reached a file-direct watcher on the second atomic save. Fixed by watching the **parent directory** instead and filtering events by filename (`is_modify() || is_create() || is_remove()`, excluding `Access` events specifically so the reload's own file read doesn't retrigger itself) — verified both saves now fire correctly.
- [x] **Fixed: Canvas node shapes ("charts") not drawing right.** Our global 2px-radius mandate (`* { border-radius: 2px !important }`) was clobbering superlore Canvas's node shapes — a `circle` node rendered as a square, `rounded`/pill nodes lost their curve. First fix attempt (`.react-flow * { border-radius: revert !important }`) silently didn't work: `revert` resets to the *user-agent* default (`0`), not to whatever author rule set the real shape radius, so it looked identical to the bug. Actual fix: exclude Canvas's shape-surface class directly at the source — `*:not(.kp-canvas-node)` in the global rule (`.kp-canvas-node` is the `SURFACE` const superlore's `nodes.mjs` puts on every shape). Verified circle/diamond/rounded node kinds render correctly now, with the flat-2px chrome elsewhere (`.page`, buttons) unaffected.
- [x] **Fixed: Canvas edges/arrows invisible in the real desktop app** (not reproducible in headless Chromium, which is all we'd verified Canvas against up to this point). Set up a real `tauri dev`-equivalent under Xvfb + WebKitGTK to test against Tauri's actual engine, and confirmed a genuine WebKitGTK-only bug: `@xyflow/react`'s own base CSS (`.react-flow .react-flow__edges svg { overflow: visible }`) never made it into superlore's compiled bundle, and WebKitGTK — unlike Chromium — force-clips the outermost `<svg>` to its own box regardless of the `overflow` property (confirmed live: computed style showed `overflow: visible`, edges still clipped). Fixed by giving the edges `<svg>` real fixed pixel dimensions (`8000px` square) so there's no box edge left to clip at. Verified against the exact multi-node/frame/routed-edge diagram from the original bug report, rendering correctly end-to-end in a real Tauri window.
- [x] **Process note**: all Canvas rendering must be checked in an actual Tauri window (or WebKit) going forward — headless Chromium/Playwright cannot catch this whole class of WebKitGTK-specific bug and had given false confidence on the two prior Canvas fixes above.
